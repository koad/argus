### Alice has no cryptographic identity

**Issue:** Alice's `id/` directory is empty — no Ed25519, ECDSA, RSA, or DSA keys generated. There is also no `trust/` directory structure. Alice cannot participate in the trust bond system.

**Why blocked:** Key generation requires running `koad-io gestate` or equivalent key generation tooling. This creates private keys that must be generated by koad (or on koad's behalf with explicit consent) and stored securely. Salus cannot generate cryptographic identity keys for another entity without koad present.

**Required action (koad):**
```bash
# On thinker, from ~/.alice:
koad-io gestate alice   # if re-gestation is safe, OR
```

**Risk:** Alice is the centerpiece of the top-of-funnel strategy. She cannot be cryptographically attributed as an author or receive trust bonds until this is resolved. Blocking trust chain participation.

### GPG revocation certificates missing from 14/15 entities

**Issue:** Only Juno has `gpg-revocation.asc` in `id/`. All other 14 entities have no key compromise recovery artifact on-disk. If a key is compromised, revocation requires generating the certificate at that moment — higher risk than having it pre-generated.

**Why blocked:** Generating a GPG revocation certificate requires access to the entity's GPG private key, which lives in the entity's GPG keyring. Salus does not have access to other entities' GPG private keys. koad must run `gpg --gen-revoke` for each entity.

**Required action (koad):**
```bash
# For each entity — run in that entity's environment or with their key:
gpg --gen-revoke <entity-key-fingerprint> > ~/.{entity}/id/gpg-revocation.asc
# Then commit to each entity's repo
```

**Affected entities (14):** vulcan, faber, chiron, vesta, muse, livy, rufus, sibyl, veritas, mercury, argus, salus, aegis, alice

**Risk:** Key compromise without a pre-generated revocation certificate requires generating the cert under adversarial conditions — higher friction, higher risk of error.

## Not an Issue

### Missing `hooks/executed-without-arguments.sh` for most entities

**Argus finding:** 7 entities (vulcan, faber, muse, veritas, mercury, alice, sibyl) have no executable hook file.

**Assessment:** This is not a gap that requires remediation. Per koad:io framework design, entities that do not need custom hook behavior rely on the framework default in `~/.koad-io/`. The hook file in an entity directory is an **override**, not a required file.

Only entities with special routing needs require a custom hook. The clearest example is Chiron's former SSH routing (which routed invocations to a remote machine). Juno has a custom hook for policy enforcement and PID locking.

For entities like Faber, Rufus, Mercury, Veritas, Muse, and Vulcan — the framework default invocation behavior is correct. Their absence of a `hooks/executed-without-arguments.sh` is intentional, not a compliance gap.

**No action required.** Argus's check flagged these as FAIL because it checks for file presence — but the correct assessment is that absence is valid for most entities. This is a false positive in the diagnostic criteria, not a real issue in the entity state.
